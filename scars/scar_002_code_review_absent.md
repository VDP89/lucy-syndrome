# scar_002 — Generated code shipping without review

## What happened (origin)

A formal audit of my own Claude Code usage returned a finding that
was uncomfortable in a specific way: **close to 100% of the code
Claude generated was being executed without me reading it first.**
My review was happening at the *functional* layer — "did the script
produce the expected output?" — and almost never at the *source*
layer. The script worked, so the script was accepted.

The consequence of skipping the source-layer review is not that
obviously broken code ships; obviously broken code fails fast and
gets caught. The consequence is that *subtly wrong* code accumulates
as technical debt the operator cannot see: performance cliffs that
only activate at larger input sizes, silent exception swallowers,
hardcoded assumptions about environment, repeated anti-patterns that
get copy-pasted into new scripts because the previous script
"worked".

The audit also surfaced a concrete pattern: the same anti-patterns
were appearing across multiple generated scripts over months. They
weren't regressions in a single codebase; they were the model
falling into the same ruts every time the task shape looked
familiar. That is Lucy Syndrome at the code-generation layer —
reading "avoid X" in a prior session's notes is not the same as
having been burned by X.

## Where it applies (trigger)

- Any script in Python or JavaScript/TypeScript generated by the
  model, over roughly 200 lines, that is about to be handed to a
  human operator for execution.
- Generators of documents (`generate_*.py`, `build_*.js`).
- Data-processing scripts: parsers, format converters, volume
  calculations, anything that touches a file format with edge cases.
- Any modification to existing code that changes more than ~50
  lines.

The hook `hook_scar_002_size.py` in this repo enforces the 200-line
threshold at the PreToolUse layer. Adjust the threshold in the hook
to match your tolerance.

## Why it gets forgotten (root cause)

- **No enforced rule.** The audit that identified the gap was a
  diagnosis, not an instruction. Nothing in the normal Claude Code
  workflow blocks unreviewed code from being handed over.
- **Speed-vs-safety tradeoff.** Reviewing 500 lines before delivery
  adds 5–15 minutes per delivery. In isolation that looks like pure
  cost.
- **Sunk-cost bias.** "The code is already written, it must be
  fine" is a fallacy the model falls into as readily as a human.
  Having generated it feels like evidence of having thought about
  it.
- **The anti-patterns are familiar.** The model has seen the "good"
  version in training data thousands of times. That familiarity
  masks the presence of the "bad" version in the output.

## Scar (fix)

**Three-step self-review before declaring code "ready to run":**

### Step 1 — Read and pattern-match

Pick five functions or sections at random from the output. Read them
end-to-end looking specifically for the following known-bad
patterns:

- **Spread-operator math on unbounded arrays.** `Math.min(...arr)`
  and `Math.max(...arr)` in JavaScript will stack-overflow or
  silently return `Infinity` when `arr.length` exceeds the engine's
  argument-count limit (around 65K in most runtimes, sometimes
  lower). For any array that could be larger than a few thousand
  elements, replace with an explicit loop. This has bitten me twice
  on datasets from industrial-measurement files.

- **Python f-strings containing JavaScript object literals.**
  Writing something like `f"const config = {js_object}"` in Python,
  where `js_object` is a dict you want rendered as JS: the curly
  braces inside the f-string get interpreted as Python
  format-spec delimiters and either throw or produce garbage.
  Either build the string with explicit concatenation, or use
  a raw string template plus `json.dumps` for the embedded object.

- **`str.replace` on files with CRLF-critical formats.** Binary or
  semi-binary formats (DXF, some CAD files, volume data files with
  fixed-offset record structures) encode line endings as part of
  their structure. Using Python's `str.replace` on these after
  reading them in text mode will corrupt the file invisibly: the
  tool that reads the file next will fail with an opaque error
  ten layers downstream. Read and write these files in binary mode,
  or use `sed` with explicit byte semantics.

- **Hardcoded Windows paths without raw strings or double escapes.**
  `"C:\Users\name\Documents"` in a Python string literal is a bug
  waiting to happen — `\U`, `\n`, `\t` etc. are valid escape
  sequences. Use raw strings (`r"..."`) or forward slashes.

- **Silent exception swallowers.** `try: ... except Exception:
  pass` and its equivalents. Every one of these is either a real
  try/except that should log or handle, or a dead branch that
  should be deleted. The pattern of "add a broad except to make
  the error go away" is a debt trap.

- **Dead imports, mock data, or dev scaffolding.** Unused imports
  at the top of a generated file, hardcoded test fixtures left in
  for manual debugging, `TODO` comments with no follow-up. Remove
  before shipping; they signal that the file was generated, not
  thought through.

### Step 2 — Verify function contracts

For each public function in the output, check that the signature
matches how its callers use it. Look for:

- Parameters in the wrong order (especially with keyword-less
  positional calls).
- Type mismatches that the language's type checker would catch but
  won't because the code runs on dynamic types.
- Ambiguous returns (a function that returns `None` on one branch
  and a data structure on another, without the caller checking).

### Step 3 — Write the review block

Before handing over the code, print a short review record:

```
SELF-REVIEW (scar_002):
- Functions reviewed: f1, f2, f3, ...
- Anti-patterns found: [list or "none"]
- Anti-patterns corrected: [list or "none"]
- Confidence: high | medium | low
```

This block is the deliverable, not a decoration. If the reviewer
(you) wants to look only at it and skip the code, they can — and
you have a record of what was or wasn't checked.

## How to verify

- The `SELF-REVIEW (scar_002)` block appears in the chat before
  every code handoff above the threshold.
- If the human operator later reports a bug matching one of the
  anti-patterns on the list → this is a **scar failure**. Log the
  failure, re-read the scar, and sharpen the pattern description
  so the next scan will catch it.

## Notes

This scar is the highest-severity one in the set because the audit
that identified it found that the absence of code review was the
single largest operational weakness of the workflow. It is also
the most expensive to apply on every invocation — self-review of
a 500-line script can take ten minutes. The trade-off is worth it
because the bugs this scar catches are the quiet kind that
accumulate into long-term technical debt if left uncaught.

The hook (`hooks/hook_scar_002_size.py`) does not perform the review
itself. It only reminds the model, at the moment a large code block
is being written, that the review protocol needs to run before the
code is handed over. The review itself is behavioral: the model
executes the three steps and writes the review block. The hook is a
context nudge, not an enforcement gate.
